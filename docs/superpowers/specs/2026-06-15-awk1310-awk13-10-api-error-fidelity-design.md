# AWK1310 detailed technical story spec

## Source story brief

`docs/tracks/agentic-workflow-kit-redesign/stories/AWK1310.md`

## Decisions resolved from the story brief

| Question | Decision | Rationale |
| --- | --- | --- |
| Central `internal/errors.ts` vs co-located error classes per module? | Add `packages/orchestrator/src/internal/errors.ts` as the single source for public API error code classes and helpers. | The error code vocabulary is shared by the public facade and can later be reused by handlers without importing the facade. A central module also avoids duplicating `code` strings. |
| Which error classes map to which existing `code` values? | `WorkflowConfigError` -> `CONFIG_INVALID`; `WorkflowTrackerError` -> `TRACKER_INVALID`; `WorkflowStoryNotEligibleError` -> `STORY_NOT_ELIGIBLE`; `WorkflowRunNotFoundError` -> `RUN_NOT_FOUND`; `WorkflowInternalError` -> `INTERNAL_ERROR`. | These preserve the existing public string values while making classification type-based. |
| Are any currently-classified errors actually retryable? | Default every class to `retryable: false`; allow the base constructor to accept `retryable: true` for future transient API failures. | Existing behavior is compatibility-first. The field becomes class-derived and meaningful without inventing a transient class outside this story's scope. |

## Exact types/contracts

Add `packages/orchestrator/src/internal/errors.ts`:

```ts
export type WorkflowApiErrorCode =
  | 'CONFIG_INVALID'
  | 'TRACKER_INVALID'
  | 'STORY_NOT_ELIGIBLE'
  | 'RUN_NOT_FOUND'
  | 'INTERNAL_ERROR';

export class WorkflowKitError extends Error {
  readonly code: WorkflowApiErrorCode;
  readonly retryable: boolean;
}
```

Concrete classes:

- `WorkflowConfigError`
- `WorkflowTrackerError`
- `WorkflowStoryNotEligibleError`
- `WorkflowRunNotFoundError`
- `WorkflowInternalError`

Helpers:

- `isWorkflowKitError(error: unknown): error is WorkflowKitError`
- `workflowKitErrorFromUnknown(error: unknown): WorkflowKitError`

`WorkflowApiErrorCode` in `api/facade.ts` must be imported from `internal/errors.ts` and re-exported
from the facade to preserve the public import path.

The public failure envelope shape does not change:

```ts
error: {
  code: WorkflowApiErrorCode;
  message: string;
  severity: 'error';
  retryable: boolean;
  details: unknown[];
  artifactRefs: WorkflowArtifactRef[];
}
```

## Query/schema/prompt/event/component design

No queries, schemas, prompts, events, routes, or components change.

Facade behavior:

- `failureEnvelope` converts unknown throwables with `workflowKitErrorFromUnknown`.
- `failureEnvelope.error.code` comes from `WorkflowKitError.code`.
- `failureEnvelope.error.retryable` comes from `WorkflowKitError.retryable`.
- `errorCodeForMessage` is removed.

Facade-owned throw sites:

- Duplicate story across tracks: throw `WorkflowTrackerError`.
- Missing story: throw `WorkflowTrackerError`.
- Multiple eligible tracks with no track target: throw `WorkflowTrackerError`.

Fallback classification for non-facade throw sites must be type/metadata based, not substring matching:

- Node `ENOENT` errors and messages generated by config loading are wrapped as `WorkflowConfigError` only at config-loading catch boundaries where the operation is resolving project config.
- Run-read handlers should wrap missing run references as `WorkflowRunNotFoundError` when catching run status/stream/inspect/report/export failures.
- Tracker validate/migrate and run-preview tracker discovery failures should wrap unknown errors as `WorkflowTrackerError` at the facade boundary.
- Any other unknown error becomes `WorkflowInternalError`.

Compatibility:

- Preserve current `code` strings exactly.
- Preserve success envelopes and `next` actions.
- Do not change CLI/MCP tool names or handler return shapes.
