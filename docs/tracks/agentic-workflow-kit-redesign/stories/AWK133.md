# AWK13.3 run-state durability and concurrency hardening story brief

not implementation-ready; create a detailed technical story spec before plan/code

## PRD criteria

| Criterion | Product outcome |
| --- | --- |
| RUN-6 | Run state stays consistent under concurrency so recoverable stopped states remain trustworthy. |
| OBS-5 | Machine-readable run artifacts (events, controls, summary, rows) are never corrupted by interleaved writes. |
| Q-2 (quality bar) | Runtime state is durable across parallel children, user interruption, and partial writes. |

## Technical solution sections

| Section | Relevance |
| --- | --- |
| Release-hardening design: AWK13.3 | Defines append serialization, malformed-line tolerance, single-writer abort, and claim CAS. |
| Data contracts: Run artifact shape | Artifact readers/writers must tolerate concurrent and partial state. |

## Dependencies

| Dependency | Reason |
| --- | --- |
| AWK13.2 | Verification/recovery changes touch `WorkflowRunner`; durability work lands on the settled shape to limit contention. |

## Scope boundary

**In scope**

- Serialize artifact appends through a single per-store promise chain (or one append handle) so
  concurrent `appendEvent`/`appendText` cannot interleave a line.
- Make every NDJSON/artifact reader malformed-line tolerant (skip or `{ raw: line }`): `readControls`,
  `readLaunchRecord`, the codex-control target reader.
- Single-source abort: `controls.ndjson` is the request log; only `WorkflowRunner` mutates
  `state.json`; out-of-process abort writes a control and is reconciled; atomic temp+rename if a
  detached run must be marked.
- Make the tracker claim atomic for `git.strategy: branch` parallel runs (compare-and-swap or
  `O_EXCL` lock), confirming `owner == self` after rename.
- Make the in-run tracker parse degrade (block the bad row) instead of throwing and killing the loop.

**Out of scope**

- New artifact files or schema versions beyond what durability requires.
- Changing abort's user-facing CLI/MCP surface (AWK13.1 owns control routing; OBS-2 surface stays).
- Validate-command parse behavior (still throws for explicit validation).

## Assumptions and blockers

| Type | Item | Evidence or resolution path |
| --- | --- | --- |
| Assumption | The malformed-line `{ raw }` pattern already used by analyzer/watch readers is the house style to extend. | Present in `analysis/runAnalyzer.ts` and watch readers. |
| Assumption | Single-writer for `state.json` is acceptable because the runner already polls `controls.ndjson`. | `applyPendingAbortControl` already reconciles control requests. |
| Blocking question | For `git.strategy: branch` parallel claims, is a lock file or in-file CAS preferred? | implement-next picks the mechanism and its failure/rollback behavior. |

## Artifact boundaries

PRD owns outcomes. Release-hardening design owns the concurrency architecture. Tracker owns
sequencing. This brief owns local scope. The detailed spec owns the serialization mechanism, reader
contracts, and claim protocol.

## Candidate surfaces

- **Files/modules:** `artifacts/FileArtifactStore.ts`, `runner/RunJournal.ts`,
  `runner/DuplicateLaunchGuard.ts`, `mcp/codexControl.ts`, `commands/handlers.ts` (abort path),
  `tracks/trackerClaimer.ts`, `tracks/markdownTracker.ts`
- **Queries/schema:** none new
- **Prompts/tools:** none
- **Events/metrics:** integrity of `events.ndjson` / `controls.ndjson`
- **Components/routes:** none

## Validation expectations

- Fault-injection tests: concurrent appends never yield a malformed line; corrupt
  `controls.ndjson` / `*.launch.json` does not break abort or duplicate detection.
- Test: two concurrent claimants yield exactly one winner.
- Test: a malformed tracker row mid-run blocks one story without aborting the run.
- `pnpm check`

## Open technical questions

| Question | Blocking? | Resolution expected in detailed spec |
| --- | --- | --- |
| Per-store promise chain vs a single long-lived append handle for serialization? | yes | Pick the mechanism that is crash-safe and testable. |
| Should out-of-process abort ever mutate `state.json`, or always go through `controls.ndjson`? | yes | Define the single-writer reconciliation rule. |
