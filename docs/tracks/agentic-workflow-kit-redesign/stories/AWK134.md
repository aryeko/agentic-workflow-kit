# AWK13.4 conservative defaults, API fidelity, and input hardening story brief

not implementation-ready; create a detailed technical story spec before plan/code

## PRD criteria

| Criterion | Product outcome |
| --- | --- |
| POL-1 | New repos default to conservative behavior with explicit approval before non-dry-run launch, enforced in the runtime, not only in skill prose. |
| TRK-2 | The capability surface reports tracker migration as available so orchestrating agents use the working path. |
| Q-10 (quality bar) | Defaults favor safety: dry-run, explicit approval, conservative preset selection, traversal-safe inputs. |

## Technical solution sections

| Section | Relevance |
| --- | --- |
| Release-hardening design: AWK13.4 | Defines the approval gate, conservative init default, capability fix, typed results, and path validation. |
| API surface: result envelopes and capabilities | Capability flags and approval results must follow the shared envelope. |

## Dependencies

| Dependency | Reason |
| --- | --- |
| AWK13.1 | The approval gate and capability surface live in handlers/config that AWK13.1 neutralizes. |

## Scope boundary

**In scope**

- Add a runtime approval/preflight gate: non-dry-run launch requires an explicit affirmative signal
  or a recorded prior dry-run+confirm; absent it, return a structured "approval required" result.
- Default `workflow-init` to `push-only` for new/unknown repos; auto-merge presets become explicit
  opt-in.
- Fix `api/facade.ts` to report `trackerMigration: true`, with a test asserting the full capability
  set.
- Type `analyzeRunHandler` as `WorkflowRunAnalysis` (not `unknown`).
- Apply repo-relative path validation to all repo-relative path config fields and the `tracksDir`
  CLI/MCP override, rejecting `..`/absolute escapes at the boundary.

**Out of scope**

- Changing preset semantics themselves (POL-2 presets stay).
- GitHub verification (AWK13.2).
- Broad refactor of handlers (AWK13.5).

## Assumptions and blockers

| Type | Item | Evidence or resolution path |
| --- | --- | --- |
| Assumption | The approval gate is a behavior change toward safety and must be documented in release notes. | Captured in AWK13.7 and AWK14. |
| Assumption | `repoRelativePath` validation (already used for `worktreeDir`) is the reusable rule for other path fields. | Present in `config/schema.ts`. |
| Blocking question | What exactly counts as "explicit approval" for MCP vs CLI callers? | implement-next defines the approval input/flag and how a prior dry-run satisfies it. |

## Artifact boundaries

PRD owns outcomes. Release-hardening design owns the safety/API design. Tracker owns sequencing. This
brief owns local scope. The detailed spec owns the approval-gate contract, init default change, and
validation rules.

## Candidate surfaces

- **Files/modules:** `commands/handlers.ts`, `skills/workflow-init/SKILL.md` + init logic,
  `config/preset.ts`, `api/facade.ts`, `config/schema.ts`, `cli/args.ts`, `mcp/tools.ts`
- **Queries/schema:** path-field validation; capability flags
- **Prompts/tools:** approval input on launch tools
- **Events/metrics:** approval decision recorded in run artifacts
- **Components/routes:** CLI/MCP launch + capability surfaces

## Validation expectations

- Test: non-dry-run launch without approval is refused with a structured result.
- Test: `workflow-init` on a fresh repo yields `push-only`.
- Test: capability set reports migration available.
- Test: `tracksDir: ../../etc` override is rejected.
- `pnpm check`

## Open technical questions

| Question | Blocking? | Resolution expected in detailed spec |
| --- | --- | --- |
| How does an MCP caller express approval without a TTY prompt? | yes | Define an explicit approval input and its envelope. |
| Should a prior dry-run in the same run directory satisfy approval, or is approval always per-launch? | no | Pick the least-surprising policy and document it. |
