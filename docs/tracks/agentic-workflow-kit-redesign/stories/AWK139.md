# AWK13.9 run-state write atomicity and durability story brief

not implementation-ready; create a detailed technical story spec before plan/code

## PRD criteria

| Criterion | Product outcome |
| --- | --- |
| RUN-6 | Recoverable stopped states stay trustworthy because run state survives a crash mid-write. |
| OBS-5 | Machine-readable full-file artifacts (`state.json`, `summary.json`) are never observed truncated. |
| Q-2 (quality bar) | Runtime state is durable across partial writes and crashed claim holders. |

## Technical solution sections

| Section | Relevance |
| --- | --- |
| Release-hardening design 2: AWK13.9 | Defines atomic temp+rename writes, stale-lock recovery, and the crash-recovery round-trip test. |
| Data contracts: Run artifact shape | Full-file artifact readers must never see a partial write. |

## Dependencies

| Dependency | Reason |
| --- | --- |
| AWK13.8 | Artifact-dir derivation moves behind the driver; atomicity lands on the settled config shape. |

## Scope boundary

**In scope**

- Make `FileArtifactStore.writeText` (hence `writeJson`) atomic via temp-file + `rename`, reusing the
  pattern in `tracks/trackerClaimer.ts` (temp write → rename → optional read-back).
- Add stale-lock recovery to the tracker claim lock (PID-liveness and/or lock-age) so a crashed
  holder cannot permanently block claimants.
- Add a crash-recovery round-trip test: write `state.json` / `*.launch.json` via the real
  `FileArtifactStore`, simulate restart, read back, assert the `RecoveryGuard` decision.

**Out of scope**

- The append path (already serialized + malformed-line tolerant in round 1 / AWK13.3).
- New artifact files or schema versions beyond what atomicity requires.
- Changing the on-disk artifact directory name (compatibility-first).

## Assumptions and blockers

| Type | Item | Evidence or resolution path |
| --- | --- | --- |
| Assumption | Temp+rename within the same directory is atomic on the supported filesystems. | Same assumption already relied on by `tracks/trackerClaimer.ts`. |
| Assumption | In-memory-only durability assertions are insufficient and should be replaced, not duplicated. | Round-2 review R2-4a; current tests assert against `MemoryArtifacts`. |
| Blocking question | Stale-lock detection via PID liveness, lock-age threshold, or both? | implement-next picks the mechanism and its reclaim/rollback behavior. |

## Artifact boundaries

PRD owns outcomes. Release-hardening design 2 owns the durability architecture. Tracker owns
sequencing. This brief owns local scope. The detailed spec owns the rename protocol, lock-recovery
rule, and test fixtures.

## Candidate surfaces

- **Files/modules:** `artifacts/FileArtifactStore.ts`, `tracks/trackerClaimer.ts`
- **Queries/schema:** none new
- **Prompts/tools:** none
- **Events/metrics:** integrity of `state.json` / `summary.json` full-file writes
- **Components/routes:** none

## Validation expectations

- Fault-injection test: a reader never observes a truncated `state.json` (sees old-complete or
  new-complete only).
- Test: a crashed claim holder's stale lock is reclaimed; a live holder's lock is respected.
- Test: crash-recovery round-trip (write via real store → restart → read → `RecoveryGuard` decision).
- `pnpm check`

## Open technical questions

| Question | Blocking? | Resolution expected in detailed spec |
| --- | --- | --- |
| Read-back verify after rename, or trust the rename? | no | Match `trackerClaimer` behavior unless a test proves read-back is needed. |
| How is a crash simulated in the round-trip test (fresh store instance vs process)? | yes | Pick a deterministic, fast simulation. |
