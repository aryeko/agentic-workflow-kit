# AWK13.1 provider-neutral driver boundary story brief

not implementation-ready; create a detailed technical story spec before plan/code

## PRD criteria

| Criterion | Product outcome |
| --- | --- |
| HC-2 | The provider-neutral driver contract holds in practice, so a future host adapter does not require redesigning runtime, recovery, config, analyzer, or control concepts. |
| HC-1 | Codex remains fully supported; existing Codex configs and tool calls keep working unchanged. |
| Q-8 (quality bar) | Product concepts avoid Codex-only coupling outside the driver package. |

## Technical solution sections

| Section | Relevance |
| --- | --- |
| Release-hardening design: AWK13.1 | Defines the neutral control/abort/error-classification additions and config aliasing. |
| Architecture and domains: Agent execution | Host interaction must live behind the `StoryRunner` contract, not leak into other domains. |

## Dependencies

| Dependency | Reason |
| --- | --- |
| AWK13 | Builds on the consolidated canonical docs and the current runtime surface. |

## Scope boundary

**In scope**

- Extend the driver contract with neutral child control (`interrupt`/`reply`), `abort`, error
  classification (`supervisionLost`/`recoverable`), and capability-downgrade description.
- Make `WorkflowRunner`, `commands/handlers.ts`, and the analyzer consume those contract methods
  instead of matching Codex error strings, calling `sendCodexInterrupt`, re-authoring Codex
  downgrade text, or hardcoding `~/.codex/sessions`.
- Move prompt rendering into a host-neutral renderer; keep host-specific phrasing inside the Codex
  driver only.
- Add a neutral child-session config namespace with `codex` retained as a back-compatible alias.
- Add neutral control tool names as aliases while keeping the existing Codex-named tools working.

**Out of scope**

- Adding a second concrete driver (future work; this story only proves the seam).
- GitHub verification (AWK13.2 owns it).
- Renaming or moving the on-disk artifact root (kept stable for compatibility).

## Assumptions and blockers

| Type | Item | Evidence or resolution path |
| --- | --- | --- |
| Assumption | Existing `.workflow/config.yaml` files must keep working via a `codex` alias onto the neutral namespace. | Rollout is compatibility-first per the technical solution and release-hardening design. |
| Assumption | The artifact root name stays as-is; only its host-meaning is removed. | It is a stable on-disk contract consumed by analyzer/inspect surfaces. |
| Blocking question | Should neutral control tools fully replace the Codex-named tools or only alias them for V1? | implement-next must choose an alias-and-keep vs deprecate path that does not break current MCP clients. |

## Artifact boundaries

PRD owns what/why. The release-hardening design owns high-level how. Tracker owns sequencing. This
brief owns local scope. The detailed technical story spec owns exact contract signatures, config
alias mapping, and prompt-renderer design.

## Candidate surfaces

- **Files/modules:** `drivers/StoryRunner.ts`, `drivers/codex-mcp/*`, `runner/WorkflowRunner.ts`,
  `commands/handlers.ts`, `mcp/codexControl.ts`, `mcp/tools.ts`, `config/schema.ts`,
  `config/configLoader.ts`, `analysis/runAnalyzer.ts`, `references/config-schema.md`
- **Queries/schema:** neutral child-session config namespace + `codex` alias
- **Prompts/tools:** host-neutral prompt renderer; neutral child-control tools
- **Events/metrics:** error-classification and capability-downgrade fields sourced from the driver
- **Components/routes:** MCP control tool surface (aliased)

## Validation expectations

- Tests asserting no `runner/`, `commands/`, `api/`, `analysis/` code branches on a host name,
  error string, tool name, or path.
- Back-compat tests: existing Codex config and Codex-named tool calls still work.
- `pnpm check`

## Open technical questions

| Question | Blocking? | Resolution expected in detailed spec |
| --- | --- | --- |
| What is the minimal driver-contract surface for control/abort/error-classification without over-fitting to Codex? | yes | Define the neutral interface and how the Codex driver implements it. |
| Should the `codex` config namespace be deprecated with a warning or kept indefinitely as an alias? | no | Pick a compatibility policy and document it. |
