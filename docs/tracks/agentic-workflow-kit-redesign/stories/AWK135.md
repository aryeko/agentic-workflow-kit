## Scope boundary

**In scope**

- **Characterization tests first:** before splitting a module, confirm its current behavior is
  covered and add any missing tests as the first step of this story. The refactor is protected by a
  safety net that exists in this PR — it does not rely on AWK13.6 (which comes later).
- Split by responsibility behind stable entry points, no behavior change: `commands/handlers.ts` →
  run-dispatch / run-control / event-normalization / run-inspection / watch; `analysis/runAnalyzer.ts`
  → parse / metrics-extraction / synthesis; extract a `ChildSupervisor` from `WorkflowRunner`'s
  `executeChild` timeout/heartbeat state machine; `tracks/markdownTracker.ts` → parse / validate /
  render.
- Bring every source file under 800 lines (target 200–400) and functions under ~50 lines where
  reasonable.

**Out of scope**

- Any behavior change, new feature, or API change (pure refactor).
- Decomposing files outside the named oversized set unless required for a clean split.
