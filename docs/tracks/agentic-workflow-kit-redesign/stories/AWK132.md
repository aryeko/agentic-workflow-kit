# AWK13.2 GitHub verification and recovery hardening story brief

not implementation-ready; create a detailed technical story spec before plan/code

## PRD criteria

| Criterion | Product outcome |
| --- | --- |
| HC-3 | PR existence/state, required-check conclusions, merge, and branch cleanup are verified against GitHub independent of child prose. |
| RUN-4 | The autonomy ceiling (implement → verify → PR → CI/review → merge → cleanup → continue) is backed by verified collaboration evidence, not self-report. |
| RUN-6 | Failed verification, stale base, merge conflict, auth failure, review uncertainty, and ambiguous child state produce recoverable stopped states from observed signals. |
| Q-5 / Q-7 (quality bars) | Outcome metrics are independently verifiable; GitHub checks/reviews/merge/cleanup are handled predictably. |

## Scope boundary

**In scope**

- Add a host-neutral collaboration-verification port with a GitHub (`gh`/Octokit) implementation:
  PR state, required-check conclusions, merge status + commit, branch-deletion, review decision.
- **Gate the irreversible action before it happens.** Under auto-merge the child must not merge on
  its own judgment: either the parent performs the merge after verification (parent-side merge), or
  merge authority is withheld until the parent emits an explicit pre-merge authorization that follows
  verification. Same for branch deletion.
- Make `CompletionGate` treat verified evidence as authority and downgrade child prose to a hint;
  keep the existing git-ancestry merge check as a second independent proof (defense-in-depth, not the
  primary control for an irreversible action).
- Fail closed when verification is unavailable (no `gh`/auth/offline): do not merge or mark merged;
  produce a recoverable stopped state naming the missing signal.
- Feed `RecoveryGuard` real `remoteBranchExists` and PR state so `safe_to_take_over` is reachable and
  the enumerated RUN-6 modes map to `manual_recovery_required` from observed signals.
- Expose capability flags distinguishing verified vs child-reported evidence in results/artifacts.

**Out of scope**

- Non-GitHub collaboration providers (future work).
- Provider-neutral contract changes beyond consuming AWK13.1 seams.
- Having the parent author PR descriptions or push code fixes; the child still opens the PR and
  pushes fixes — only the irreversible merge/branch-delete is gated by parent verification.

## Assumptions and blockers

| Type | Item | Evidence or resolution path |
| --- | --- | --- |
| Assumption | `gh` is the V1 verification transport; it is already a documented dependency. | Config/docs already assume GitHub CLI for CI/review waits. |
| Assumption | Verification-unavailable must fail closed, never spuriously fail a genuinely passing run. | Q-10 safety bar and release-hardening principle 2/3. |
| Blocking question | Which required-check semantics define "CI passed" — all checks, required checks only, or branch-protection-defined? | implement-next must pick a rule that matches GitHub branch protection. |

## Artifact boundaries

PRD owns outcomes. Release-hardening design owns the verification architecture. Tracker owns
sequencing. This brief owns local scope. The detailed spec owns the port interface, evidence schema,
and gate/recovery state transitions.

- **Files/modules:** new collaboration-inspector module, `runner/CompletionGate.ts`,

- **Events/metrics:** verified check/review/merge/cleanup evidence + verification-availability flag
- **Components/routes:** capability flags on CLI/MCP results

## Validation expectations

- Test: with auto-merge enabled and CI/review unverified or failing, no merge or branch deletion
  occurs (the merge is withheld) — not merely that the run is marked incomplete afterward.
- Integration test: a child claiming a fake merge / green CI where GitHub disagrees yields a
  recoverable stopped state.
- Test: verification-unavailable fails closed with a named missing signal.
- Real-git adapter tests (`isCommitReachableFromRef` / `readFileFromRef` / `refreshBaseBranch`) and
  the non-ancestor merge rejection land here, with this story, not in a later test story.
- Test: `RecoveryGuard` returns `safe_to_take_over` for a verified-clean fixture and
  `manual_recovery_required` per RUN-6 mode.
- `pnpm check`
