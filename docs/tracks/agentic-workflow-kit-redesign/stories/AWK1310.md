# AWK13.10 public API error fidelity story brief

not implementation-ready; create a detailed technical story spec before plan/code

## PRD criteria

| Criterion | Product outcome |
| --- | --- |
| Q-5 (quality bar) | API consumers can rely on a stable error contract rather than parsing prose. |
| API surface | The result envelope's `code`/`retryable` fields are meaningful and stable. |

| Section | Relevance |

| Release-hardening design 2: AWK13.10 | Defines typed error classes feeding the facade's `code`/`retryable`. |

## Dependencies

| Dependency | Reason |
| --- | --- |

## Scope boundary

**In scope**

- Replace `errorCodeForMessage` substring-matching in `api/facade.ts` with typed error classes
  carrying a stable `code` and a real `retryable` value.
- Raise typed errors at the throw sites that currently produce classified errors; the facade reads
  `error.code` instead of matching message text.
- Preserve existing `code` string values and envelope shape (compatibility-first).

**Out of scope**

- New error codes or envelope fields beyond making `retryable` meaningful.
- Changing CLI/MCP tool surfaces or success-path payloads.

## Assumptions and blockers

| Type | Item | Evidence or resolution path |
| --- | --- | --- |
| Assumption | The current `code` string values are a stable contract to preserve. | Round-2 review R2-5; existing consumers read them. |
| Assumption | `retryable` should reflect the error class, not be hardcoded `false`. | `api/facade.ts:537` hardcodes `false`. |
| Blocking question | Central `internal/errors.ts` vs co-located error classes per module? | implement-next picks placement; keep one source of `code` values. |

## Artifact boundaries

PRD owns outcomes. Release-hardening design 2 owns the typed-error approach. Tracker owns sequencing.
This brief owns local scope. The detailed spec owns the error-class hierarchy and the throw-site map.

## Candidate surfaces

- **Files/modules:** `api/facade.ts`, `internal/errors.ts` (new) or `internal/guards.ts`, throw sites
  feeding the facade
- **Queries/schema:** none new
- **Prompts/tools:** none
- **Events/metrics:** none
- **Components/routes:** none

## Validation expectations

- Test: error `code` is derived from error type; rewording a human-readable message does not change
  any `code`.
- Test: `retryable` reflects the actual error class.
- Test: the full advertised error-code set is exercised.
- `pnpm check`

| Are any currently-classified errors actually retryable? | no | Default `retryable: false`; flag transient classes explicitly. |
