# AWK13.14 Budget and telemetry fidelity story brief

not implementation-ready; create a detailed technical story spec before plan/code

## PRD criteria

| Criterion | Product outcome |
| --- | --- |
| OBS-4 | Per-run failed-tool-call data is recorded when the host exposes it, not advertised then left null. |
| POL-4 | Configurable budget dimensions map to metrics the runtime can actually enforce. |
| POL-5 | A configured `failedToolCalls` budget either trips on real data or is not offered. |
| Q-5 (quality bar) | Observability data is enough to verify outcome metrics without phantom dimensions. |

## Technical solution sections

| Section | Relevance |
| --- | --- |
| Release-hardening design 3: AWK13.14 | Defines the wire-it-or-retire-it decision for `failedToolCalls`. |
| Data contracts | Runtime artifact / budget shapes. |
| AI/observability/operations | Session-log metric extraction. |

## Dependencies

| Dependency | Reason |
| --- | --- |
| AWK13.13 | `RunJournal` lives in `runner/`; lands after the runner reshape so metric wiring sits on the final shape. |

## Scope boundary

**In scope**

- Resolve the `failedToolCalls` mismatch (`runner/RunJournal.ts:305` always null vs POL-4 advertising
  it as a budget dimension) by **either** deriving the metric from session-log/tool-activity parsing so
  the budget can trip, **or** retiring the dimension from config and documenting the limitation.
- Update the runtime-artifact/budget contract and flag the AWK14 release note accordingly.

**Out of scope**

- Enabling live token telemetry (`tokenTelemetryLive` stays `false`; honest null-with-reason kept).
- Structured-output enforcement (`structuredOutputEnforced` stays `false`; AWK14 release-note item).
- Runner type/structure changes (AWK13.13).

## Assumptions and blockers

| Type | Item | Evidence or resolution path |
| --- | --- | --- |
| Blocking question | Does the Codex session-log expose failed tool calls in a parseable form? | Detailed spec inspects the log shape used by `drivers/codex-mcp/evidenceParser.ts`; if yes, wire it; if no, retire the dimension. |
| Assumption | `metrics/availability.ts` null-with-reason machinery stays for genuinely unavailable runs. | Already the honest fallback at HEAD. |

## Artifact boundaries

PRD owns outcomes. Release-hardening design 3 owns the decision frame. Tracker owns sequencing. This
brief owns local scope. The detailed spec owns the parsing/derivation or the schema removal.

## Candidate surfaces

- **Files/modules:** `runner/RunJournal.ts`, `metrics/` (budget + availability), `config/schema.ts`,
  possibly `drivers/codex-mcp/evidenceParser.ts`
- **Queries/schema:** `config/schema.ts` budget dimensions
- **Prompts/tools:** none
- **Events/metrics:** `summary.json`, `budgets.json` `failedToolCalls`
- **Components/routes:** none
- **Docs:** `references/runtime-artifact-contract.md`

## Validation expectations

- A `failedToolCalls` budget either trips on real failed-tool-call counts, or the dimension is no longer
  accepted by config and the limitation is documented.
- No config field advertises a structurally inert budget control.
- `pnpm check`
