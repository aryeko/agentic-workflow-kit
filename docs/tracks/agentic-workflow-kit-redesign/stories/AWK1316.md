# AWK13.16 Codex child-session speed policy story brief

not implementation-ready; create a detailed technical story spec before plan/code

## PRD criteria

| Criterion | Product outcome |
| --- | --- |
| POL-3 | Users can configure speed/service-tier policy without dropping into raw Codex config. |
| Q-4 (quality bar) | Agent launch policy includes model, reasoning, speed/service tier, permissions, and budget. |
| HC-1 | Codex Fast and Standard launch behavior is fully supported in V1 child sessions. |
| HC-2 | Product config remains provider-neutral; Codex service-tier quirks stay behind the driver. |

## Technical solution sections

| Section | Relevance |
| --- | --- |
| Domain model | Defines speed/service-tier as part of child-session launch policy. |
| Quality bars / acceptance criteria | Adds speed/service-tier policy to POL-3 and Q-4. |
| Architecture and domains | Places speed resolution in child-session launch policy. |
| Data contracts | Defines normalized `childSession.speed` and Codex mapping. |

## Dependencies

| Dependency | Reason |
| --- | --- |
| AWK13.15 | Lands after coverage/test DevX headroom so the public config-schema addition can add tests without immediately destabilizing the ratchet. |

## Scope boundary

**In scope**

- Add first-class child-session speed policy with values `derive`, `fast`, and `standard`, defaulting to
  `derive`.
- Map the normalized policy in the Codex MCP driver:
  - `derive`: pass no service-tier override and let Codex/global config decide.
  - `fast`: pass `service_tier: fast` and enable the Codex Fast-mode feature.
  - `standard`: pass explicit service-tier clear/null and `notice.fast_default_opt_out: true`.
- Preserve raw `childSession.config` pass-through for advanced driver-specific config while rejecting or
  clearly resolving conflicts with the normalized speed policy.
- Update config schema, generated JSON schema, config docs, presets/examples if needed, package docs, and
  plugin fixture copies.

**Out of scope**

- Adding a second concrete driver.
- Changing model or reasoning defaults.
- Making MCP child sessions appear as Codex app chat-list threads.
- Changing Codex billing or credit accounting; this story only controls launch configuration.

## Assumptions and blockers

| Type | Item | Evidence or resolution path |
| --- | --- |
| Assumption | Codex Standard is not the literal service-tier value `"standard"`; it is explicit clear/null plus the Fast-default opt-out marker. | Detailed spec should encode the Codex mapping before implementation. |
| Assumption | `derive` should be the default to preserve existing configs and global Codex user preferences. | Compatibility-first release-hardening principle. |
| Blocking question | Should speed be global-only under `childSession.speed`, or also overridable per agent profile? | Detailed spec chooses the smallest config surface that meets POL-3. Recommended default: global now, profile override only if implementation shows an immediate need. |
| Blocking question | On conflict between `childSession.speed` and raw `childSession.config.service_tier`, reject validation or let normalized speed win? | Recommended default: validation rejects ambiguous conflicts so cost/routing intent is explicit. |

## Artifact boundaries

PRD owns the launch-policy outcome. Technical solution owns the normalized config model. Tracker owns
sequencing. This brief owns local scope. The detailed spec owns exact schema shape, compatibility
rules, and Codex tool-input mapping.

- **Files/modules:** `packages/orchestrator/src/config/schema.ts`,
  `packages/orchestrator/src/config/configLoader.ts`,

## Validation expectations

- Config-schema tests cover `derive`, `fast`, `standard`, defaulting, and conflict handling.
- Codex tool-input tests assert exact config mapping for all three policies, including `standard`
  emitting clear/null plus `notice.fast_default_opt_out: true`.
- Back-compat tests prove existing `childSession.config.service_tier` configs still work when no
  normalized speed policy is set.
- Docs/config sync tests pass, including generated schema drift checks and plugin fixture sync.
- `pnpm check`

## Open technical questions

| Question | Blocking? | Resolution expected in detailed spec |
| --- | --- | --- |
| Global-only `childSession.speed` vs per-profile override? | yes | Pick the smallest compatible surface; document future extension if deferred. |
| Reject conflict vs normalized speed wins over raw service-tier keys? | yes | Pick an unambiguous behavior and test it. |
| Should launch artifacts record resolved speed separately from raw Codex config? | no | Prefer recording normalized speed if the resolved child-session shape already has a home for it. |
